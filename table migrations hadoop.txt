Hive table -> HDFS directory of Avro/Parquet files -> DistCp to cloud object storage -> register/rewrite in Databricks


History load: DistCp older partitions / folders from HDFS to cloud storage in parallel.
Catch-up sync: Re-run DistCp incrementally for the latest partitions while Hadoop is still active.
Streaming cutover: Start Databricks ingestion from the live source, validate parity, then switch producers/consumers.

A practical pattern is:

copy old partitions first

validate counts / file sizes / partition completeness

copy hot partitions more frequently

leave the most recent active partitions for the final sync window

Extract Hive metadata and Sentry access model.

Grant DistCp read access to HDFS Hive-managed table path and cloud write access to landing storage.

DistCp historical Parquet partitions from oldest to newest into a raw external landing path. DistCp is designed for this kind of large distributed copy.

In Databricks, create Bronze Delta as the target lakehouse layer.

Start the new Databricks streaming pipeline writing directly to Bronze Delta, not to the raw Parquet landing path.

Continue incremental/final sync of the latest historical HDFS partitions into the raw landing area.

Load/reconcile the final historical gap into Bronze Delta. For Parquet, use CONVERT TO DELTA only on stable paths, because writes arriving after conversion starts may not be reflected.

Build/rebuild Silver from Bronze and Gold from Silver.

Apply and validate Unity Catalog permissions before consumer cutover. Unity Catalog supports CONVERT TO DELTA for Parquet in external locations it manages.

Cut over readers/writers to Databricks.

Start Databricks Bronze streaming.

Load final historical gap into Bronze Delta.

Validate counts and watermarks.

Enable Silver/Gold jobs in Databricks.

Grant final permissions.

Switch consumers to Databricks tables.

So for your Bronze history in HBase, the safest design is:

Start Kafka → Databricks Bronze at T0

Take an HBase snapshot around the same boundary for consistency

Backfill HBase history < T0 into Bronze Delta

Deduplicate on a stable business/message key

Keep Kafka as the live path; stop relying on HBase for new Bronze after cutover

My recommended end-to-end plan
Phase 1: preparation

Create Databricks target tables first:

main.bronze.raw_messages

main.silver.curated_events

main.gold.standardized_metrics

Also define the cutover timestamp T0, the business key, and the event-time column for each layer.

-------------------
Phase 2: start the new live path

Start Kafka → Databricks Bronze at T0. Do not keep pushing new raw data into Hadoop as the primary path after that point.

------------------------------------

Phase 3: historical Bronze from HBase

For HBase, do two things in parallel:

take a snapshot for a consistent point-in-time copy / rollback anchor

run a bounded historical extraction from HBase for records < T0 into Bronze Delta

The snapshot is your safety and consistency anchor because HBase snapshots are immutable and low-impact; exporting a snapshot copies HFiles, logs, and snapshot metadata to the destination filesystem.
--------------
Phase 4: historical Silver from HDFS

Move curated HDFS history into Databricks Silver.

If it is Parquet and stable, use in-place conversion or clone-oriented migration

If it is Avro or another format, land and rewrite into Delta

Databricks explicitly supports CONVERT TO DELTA for Parquet and recommends CLONE if the source keeps receiving updates during migration.

Phase 5: historical Gold from HDFS

Do the same for Gold history from standardized HDFS into Databricks Gold.

Phase 6: final reconciliation

Validate row counts, time ranges, and duplicates across all three target layers, then stop the old Hadoop-side flow.
